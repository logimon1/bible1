# 납품용 제품 품질 감사 - 2026-07-13

## 결론

현재 제품은 통제된 데모 또는 소규모 파일럿에는 사용할 수 있지만, 그대로 판매·납품할 production 제품은 아닙니다.

- 앱: 기본 자동검사는 통과하지만 참가자 권한, 보상 경제, PostgreSQL 저장 구조에 차단 이슈가 있습니다.
- 인쇄물: QR·카드·운영표의 골격은 있으나 인쇄소용 도련·재단 규격과 최종 URL이 반영되지 않았습니다.
- 교재: 학생용 2쪽, 교사용 1쪽의 운영 워크시트 수준으로, 공과책 원고와 수업 자료가 아직 없습니다.
- 콘텐츠: 앱 설정과 인쇄 패키지가 서로 다른 미션 원본을 사용해 월간 모드에서 누락이 발생할 수 있습니다.

## 실행한 검사

### 앱

- `npm.cmd run doctor`: 통과. `ADMIN_PIN`, `DATABASE_URL`, `DELIVERY_BASE_URL` 미설정 경고.
- `npm.cmd run check`: 통과.
- `npm.cmd run smoke`: 통과.
- `npm.cmd audit --omit=dev`: 알려진 취약점 0건.
- 모바일 390px 실제 화면과 주요 진입 경로 확인.

### 인쇄 패키지

- `npm.cmd run verify`: 통과.
- PDF 6종을 Poppler로 렌더해 페이지 수와 배치를 확인.
- 자동 검증은 파일 존재와 최소 크기를 확인할 뿐, 교재 내용·도련·재단선·QR 실주소·수업 가능성을 판정하지 않습니다.

## 앱 차단 이슈

### P0. 참가자 ID만 알면 다른 참가자로 행동할 수 있음

공개 상태·랭킹에 참가자 ID가 노출되고, 상태 조회·뽑기·QR 보상·교환 API는 재접속 코드 없이 `playerId`를 신뢰합니다. 참가자별 세션 또는 서명된 토큰이 필요하며, 공개 응답에서 내부 ID와 불필요한 개인정보를 제거해야 합니다.

근거:

- `delivery/app/server/core.js`의 랭킹·상태 응답
- `delivery/app/server/api.js`의 참가자 상태, 뽑기, QR, 교환 액션

### P0. 무료 뽑기를 무제한 반복할 수 있음

학생 화면에서 1·2·3회 뽑기를 언제든 호출할 수 있고 서버가 티켓·비용·미션 권한을 확인하지 않습니다. QR 보상과 랭킹의 의미가 사라집니다. 서버가 보유 뽑기권을 원자적으로 차감하도록 바꿔야 합니다.

### P0. PostgreSQL 전체 초기화가 실제 삭제를 보장하지 않음

메모리 데이터는 비우지만 저장 단계에서 기존 참가자·완료 기록·로그를 삭제하지 않아 다음 요청에서 복원될 수 있습니다. 운영 단위별 transaction 안에서 명시적으로 삭제하고 통합 테스트해야 합니다.

### P0/P1. 조회도 전체 DB 로드·전체 재저장·전역 잠금을 수행

현재 PostgreSQL 경로는 요청마다 Pool을 만들고 전체 테이블을 읽은 뒤 전체 상태를 다시 저장합니다. 클라이언트의 3.5~5초 폴링과 참가자 80명 설정을 합치면 지연과 타임아웃 위험이 큽니다. API별 SQL CRUD, 프로그램 단위 조건, connection pool 재사용으로 재설계해야 합니다.

### P1. 월간 QR 범위가 설정과 다름

인쇄 데이터는 공통 미션을 `both`로 보지만 앱 설정은 같은 미션 일부를 `retreat`로 둡니다. 앱 인쇄는 현재 모드와 정확히 같은 값만 남겨 월간 핵심 QR이 빠질 수 있습니다. 단일 콘텐츠 원본과 `mode in [both, current]` 규칙이 필요합니다.

### P1. 히든 QR과 관리자 인쇄 경로 보호 부족

공개 상태 응답과 PIN 없는 인쇄 경로를 통해 히든 QR 정보가 노출될 수 있습니다. 히든 코드·인쇄 페이지는 관리자 인증 뒤에만 제공해야 합니다.

### P1. 교회별 데이터 격리 없음

스키마에 프로그램 ID가 있어도 저장소 조회가 프로그램별로 제한되지 않습니다. 첫 상품은 교회별 전용 배포와 교회별 전용 DB로 한정하고, 공유 SaaS는 tenant 격리와 권한 모델을 별도 설계한 뒤 진행합니다.

### P1. 현재 납품용 폴더가 Git에 추적되지 않음

`delivery/app`과 `delivery/print-package`가 현재 작업 트리에서 새 폴더로만 표시됩니다. 이 상태에서는 현재 커밋 기준 CI, 복구, 배포 재현이 불가능합니다. 기존 루트 구조를 두 활성 영역으로 옮기는 사용자 변경 전체를 검토한 뒤 의도적으로 커밋해야 합니다.

## 앱 품질 보강 이슈

- `check`는 JavaScript 문법 검사이며, `test`는 파일 DB 스모크 테스트의 별칭입니다. 실제 PostgreSQL 초기화·동시성·부하 테스트가 없습니다.
- 관리자 PIN과 재접속 코드에 시도 횟수 제한이나 잠금이 없습니다.
- 요청 본문 크기 제한이 없습니다.
- CSS·JS·PNG까지 `Cache-Control: no-store`라 큰 이미지 자산을 반복 내려받습니다.
- 팀명이 자유 입력이라 `1조`, `1 조`처럼 같은 팀이 나뉠 수 있습니다.
- 390px 모바일 화면에서 상단 행사명이 한 줄로 잘리고 이름·조 입력 폭이 지나치게 압축됩니다. 모바일 첫 화면의 정보 우선순위와 반응형 규칙을 다시 잡아야 합니다.

- 종이 교정본의 QR 스캔·색상·재단 테스트

## 권장 제품 방향

1. 첫 상품을 `중·고등부 4주 전신갑주 공과 세트`로 한정합니다.
2. 월간 4주판을 콘텐츠 원본으로 만들고, 수련회판은 1~3일 압축 파생판으로 만듭니다.
3. 교회별 전용 앱 + 교회별 전용 DB 방식으로 먼저 판매합니다.
4. 앱과 인쇄물은 하나의 커리큘럼 JSON/스키마에서 생성하도록 통합합니다.
5. 앱 P0 수정과 실제 DB 부하 테스트가 끝나기 전에는 production 납품 판정을 하지 않습니다.
